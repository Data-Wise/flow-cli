    Spring-Only Workflow¶

    Annual Cycle¶

    Spring-only teaching dramatically simplifies workflow:

    January-May: SPRING 2026 ACTIVE
‚îú‚îÄ‚îÄ High activity (daily commits)
‚îú‚îÄ‚îÄ Weekly deployments
‚îú‚îÄ‚îÄ draft ‚Üí production (frequent merges)
‚îî‚îÄ‚îÄ Student-facing production branch

June-December: BETWEEN SEMESTERS (9 months!)
‚îú‚îÄ‚îÄ Major content overhaul
‚îú‚îÄ‚îÄ All work on draft branch only
‚îú‚îÄ‚îÄ No deployment pressure
‚îú‚îÄ‚îÄ No parallel semester to manage
‚îî‚îÄ‚îÄ Prepare Spring 2027

January 10-12: TRANSITION DAY
‚îú‚îÄ‚îÄ Merge draft ‚Üí production
‚îú‚îÄ‚îÄ Tag spring-2026-final
‚îú‚îÄ‚îÄ Tag spring-2027-start
‚îî‚îÄ‚îÄ Spring 2027 begins

    Key Simplifications:

    ‚úÖ No Future Semester Branch Needed - Never teaching Fall and Spring simultaneously - Don't need spring-2027 branch during Spring 2026 - All future work stays on draft

    ‚úÖ Extended Prep Period (9 months) - Summer: Rest, reflect, plan (3 months) - Fall: Intensive content creation (4 months) - Winter: Final polish (1 month) - No rush, no pressure

    ‚úÖ Relaxed Between-Semester Work - Experiment freely - Rewrite entire sections - Change technology - Test new tools

    ‚úÖ Simplified Branch Strategy - During semester: draft ‚Üí production (frequent) - Between semesters: draft only (no merges)

    Branch Architecture¶

    Structure¶

    draft              # Your daily workspace
‚îú‚îÄ‚îÄ All commits here first
‚îú‚îÄ‚îÄ Push freely for backup
‚îî‚îÄ‚îÄ Experimental, unfinished content OK

production         # Students see this
‚îú‚îÄ‚îÄ Only updated via explicit merge
‚îú‚îÄ‚îÄ Stable, tested content only
‚îî‚îÄ‚îÄ Deploys automatically to GitHub Pages

Tags (no branches)
‚îú‚îÄ‚îÄ spring-2025-final
‚îú‚îÄ‚îÄ spring-2026-final
‚îî‚îÄ‚îÄ spring-2027-final

    Mental Model¶

    draft = Unrestricted workspace - Commit anything, anytime - Push for backup (no deployment) - Can break things safely

    production = Locked gate - Only merge when ready - Students see immediately - Must be working content

    Tags = Semester snapshots - Permanent historical record - Can always recover old versions - Branches deleted after tagging

        uses: quarto-dev/quarto-actions/setup@v2

          publish_dir: ./_site
