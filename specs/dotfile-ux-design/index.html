    Dotfile Management UX Design for flow-cli¶

    Date: 2026-01-08 Version: 1.0 Status: Design Proposal

    Executive Summary¶

    This document proposes UX integration for dotfile management (chezmoi + Bitwarden) into flow-cli. The design follows flow-cli's ADHD-friendly principles: discoverable, consistent, forgiving, fast.

    Recommendation: Use dot dispatcher pattern with 3 core actions: edit, sync, status

    1. Command Naming Analysis¶

    Option A: dot (Dotfiles) - RECOMMENDED¶

    Pros: - Short, memorable (2 letters like cc, wt, tm) - Domain-specific (dotfile = df) - Available (no conflicts with common commands) - Follows dispatcher naming pattern

    Cons: - Minor conflict with GNU coreutils dot (disk free) - Mitigation: Users typically use full /usr/bin/df or disk free alternatives (duf, dust) - flow-cli users likely prefer modern alternatives already installed via Brewfile

    Example commands:

    df              # Status overview (default)
df edit .zshrc  # Edit dotfile
df sync         # Pull latest changes
df push         # Push local changes
df status       # Show sync status

    Option B: dot (Explicit)¶

    Pros: - Clear, unambiguous - No conflicts

    Cons: - 3 characters (longer than flow-cli convention) - Less ergonomic for daily use - Graphviz users might find minor conflict

    Option C: config (Generic)¶

    Pros: - Self-documenting

    Cons: - 6 characters (too long for ADHD-friendly quick commands) - Conflicts with many tools (git config, npm config, etc.) - Too generic (doesn't specify dotfiles vs app configs)

    Decision: Use dot dispatcher

    2. Dispatcher Architecture¶

    Pattern: dot <action> [target] [options]¶

    Following established flow-cli dispatcher patterns (g, mcp, cc, wt):

    df() {
    case "$1" in
        # Core workflows
        edit|e)     _dot_edit "$@" ;;
        sync|pull)  _dot_sync "$@" ;;
        push|up)    _dot_push "$@" ;;
        apply)      _dot_apply "$@" ;;

        # Status & info
        status|st)  _dot_status "$@" ;;
        diff|d)     _dot_diff "$@" ;;
        list|ls)    _dot_list "$@" ;;

        # Secret management
        secret|sec) _dot_secret "$@" ;;
        unlock)     _dot_unlock "$@" ;;

        # Setup & troubleshooting
        init)       _dot_init "$@" ;;
        doctor)     _dot_doctor "$@" ;;

        # Help
        help|h)     _dot_help ;;

        # Default: status
        "")         _dot_status ;;
        *)          echo "df: unknown action: $1" >&2; return 1 ;;
    esac
}

    3. The 3 Most Frequent Operations¶

    Based on dotfile workflow analysis:

    1. dot edit <file> - Quick Edit (60% of daily use)¶

    Flow:

    df edit .zshrc
# Opens in chezmoi: chezmoi edit ~/.zshrc
# User makes changes
# Auto-preview on save
# Prompt: Apply changes? [Y/n]

    ADHD-friendly features: - One command, no mental overhead - Preview changes before applying - Smart default (apply = yes) - Undo available via dot diff + dot apply --force

    Smart path resolution:

    df edit .zshrc     ‚Üí ~/.config/zsh/.zshrc
df edit zshrc      ‚Üí ~/.config/zsh/.zshrc (fuzzy match)
df edit git        ‚Üí ~/.gitconfig (intelligent match)
df edit ssh        ‚Üí ~/.ssh/config

    2. dot sync - Pull Latest (25% of daily use)¶

    Flow:

    df sync
# Pull from remote ‚Üí preview changes ‚Üí apply
# Output:
# ‚úì Pulled latest from origin/main
# üìù Changes to apply:
#    M ~/.zshrc (3 lines changed)
#    M ~/.gitconfig (1 line added)
# Apply now? [Y/n]

    Safety features: - Shows diff before applying - Backs up current files - Rollback available: dot undo

    3. dot status - Check Sync State (15% of daily use)¶

    Flow:

    df
# OR: df status

# Output:
# üì¶ Dotfiles Status
#
# üü¢ Synced (last: 2h ago)
# üìç Machine: iMac.local
# üîê Bitwarden: unlocked
#
# Tracked files: 12
# Modified: 0
# Behind remote: 0 commits
# Secrets: 3 injected
#
# üí° Next: df sync (to pull latest)

    Information hierarchy: 1. Sync state (most critical) 2. Machine context (helps with multi-device) 3. Security status (peace of mind) 4. File counts (overview) 5. Suggested action (ADHD-friendly next step)

    4. Error Recovery Design¶

    Problem 1: Bitwarden Session Expired¶

    UX Flow:

    $ df edit .zshrc
‚ö† Bitwarden session expired
üîì Unlock now? [Y/n] y
üîë Enter master password: ********
‚úì Unlocked Bitwarden
‚úì Opening ~/.zshrc in $EDITOR...

    Auto-recovery features: - Detects expired session before opening editor - One-step unlock prompt - Remembers intended action after unlock - Falls back to no-secret mode if user declines

    Manual unlock:

    df unlock
# OR
df secret unlock

    Problem 2: Chezmoi Merge Conflicts¶

    UX Flow:

    $ df sync
‚ö† Merge conflict detected:
   ~/.zshrc (local changes vs remote changes)

Options:
  1) Keep local changes (discard remote)
  2) Keep remote changes (discard local)
  3) Manual merge (open in editor)
  4) Abort (stay in current state)

Choice [1-4]: 3

‚úì Opening merge editor...
(After user resolves)
‚úì Conflict resolved
Apply merged version? [Y/n]

    Safety mechanisms: - Always shows both versions before merge - Backup created automatically - Undo available: dot undo - Merge help: dot doctor conflict

    Problem 3: Secret Injection Failure¶

    UX Flow:

    $ df apply
‚ö† Secret injection failed: API key not found in Bitwarden
‚ùì Item name: "Desktop Commander API"

Troubleshoot:
  1) df secret list (see all secrets)
  2) df secret add (add missing secret)
  3) df doctor (diagnose issues)

Proceed without secrets? [y/N] n
‚úó Aborted. Fix secrets first.

    Graceful degradation: - Clear error message with exact item name - Actionable suggestions - Option to skip secrets temporarily - Doctor command for diagnosis

    8. Smart Defaults & Forgiveness¶

    Smart Path Resolution¶

    df edit zshrc       # ‚Üí ~/.config/zsh/.zshrc
df edit .zshrc      # ‚Üí ~/.config/zsh/.zshrc
df edit git         # ‚Üí ~/.gitconfig
df edit gitconfig   # ‚Üí ~/.gitconfig
df edit ssh         # ‚Üí ~/.ssh/config

    Fuzzy matching logic: 1. Exact match in tracked files 2. Basename match (e.g., "zshrc" ‚Üí ".zshrc") 3. Substring match (e.g., "git" ‚Üí ".gitconfig") 4. If multiple matches: show picker

    Safe Defaults¶

    • dot sync ‚Üí Preview before applying (default: yes)
    • dot push ‚Üí Show diff before pushing (default: yes)
    • dot edit ‚Üí Preview changes after save (default: apply)
    • dot apply ‚Üí Create backup before applying

    Undo Support¶

    df undo             # Undo last apply
df undo --list      # Show undo history (last 10)
df undo 3           # Undo to 3 commits ago

    Implementation: Leverage chezmoi's backup system

    Dry Run Mode¶

    df sync --dry-run   # Preview without applying
df push -n          # Preview without pushing
df apply --dry      # Show what would change

    12. Alternatives Considered¶

    Option: Integrate into flow command¶

    flow dotfile edit
flow dotfile sync

    Rejected because: - flow command is meta-level (flow test, flow build, flow doctor) - Dotfiles are domain-specific (like git, mcp, obs) - Dispatcher pattern is more discoverable

    Option: Separate chezmoi wrapper¶

    cm edit .zshrc
cm sync

    Rejected because: - Not integrated with flow-cli ecosystem - Another command to remember - Loses dash/work integration

    Option: No integration, use chezmoi directly¶

    Rejected because: - Chezmoi UX is not ADHD-optimized - No integration with flow-cli workflows - More cognitive load for users
