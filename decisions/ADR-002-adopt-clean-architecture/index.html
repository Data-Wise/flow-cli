    ADR-002: Adopt Clean Architecture for Long-Term Maintainability¶

    Status: âœ… Accepted Date: 2025-12-20 Deciders: Development Team Technical Story: Planning for Week 2+ Architecture Refactor

    Context and Problem Statement¶

    The flow-cli system currently uses a simple 3-layer architecture (Frontend â†’ Backend â†’ Vendor) that works well for the initial implementation. As the system grows, we need to decide on a more robust architectural pattern that will:

    • Support testing without extensive mocking
    • Allow swapping implementations (e.g., file system â†’ database)
    • Keep business logic independent of frameworks
    • Enable multiple interfaces (CLI, Desktop UI, REST API)

    Key Question: Should we continue with the current ad-hoc structure, or adopt a formal architecture pattern like Clean Architecture, Hexagonal Architecture, or Layered Architecture?

    Decision Drivers¶

    • Testability - Must be able to test business logic without I/O
    • Flexibility - Should easily swap data sources (files â†’ DB â†’ API)
    • Independence - Core logic shouldn't depend on frameworks
    • Maintainability - New developers should understand structure quickly
    • Scalability - Architecture should support growth (desktop app, web UI)
    • ADHD-Friendly - Clear boundaries reduce cognitive load

    Considered Options¶

    Option 1: Continue Current 3-Layer Architecture¶

    Frontend (ZSH) â†’ Backend (Node.js) â†’ Vendor (Shell)

    Pros:

    • âœ… Simple and straightforward
    • âœ… Easy to implement initially
    • âœ… Minimal files and boilerplate

    Cons:

    • âŒ Business logic mixed with infrastructure
    • âŒ Hard to test (file system dependencies)
    • âŒ Difficult to swap implementations
    • âŒ Controllers know too much about data access

    Option 2: Layered Architecture¶

    Presentation â†’ Business Logic â†’ Data Access

    Pros:

    • âœ… Well-understood pattern
    • âœ… Clear separation of concerns
    • âœ… Better than current state

    Cons:

    • âŒ Allows dependencies to flow downward only (not ideal)
    • âŒ Database layer often becomes coupled to business logic
    • âŒ Doesn't enforce dependency inversion

    • âœ… Well-documented pattern (books, articles, examples)

    Decision Outcome¶

    Chosen option: "Clean Architecture + Hexagonal Architecture" (Option 3)

    Rationale¶

    1. Testability is Critical The current system is hard to test because business logic is intertwined with file I/O. Clean Architecture separates these concerns completely:
    // BEFORE: Hard to test
async function createSession(project) {
  const path = `~/.config/zsh/.worklog`
  await fs.writeFile(path, JSON.stringify({ project }))
}

// AFTER: Easy to test
class CreateSessionUseCase {
  constructor(sessionRepository) {
    this.repo = sessionRepository // Interface, not file system
  }

  async execute({ project }) {
    const session = new Session(id, project)
    await this.repo.save(session)
    return session
  }
}

// Test with mock repository (no file I/O)
const mockRepo = { save: async s => s }
const useCase = new CreateSessionUseCase(mockRepo)
    1. Future-Proofing for Multiple Interfaces We plan to add:
    2. Desktop UI (Electron) - in progress
    3. REST API (future)
    4. Web dashboard (future)

    Clean Architecture makes this trivial - same use cases, different controllers:

    // Same use case, different controllers
const useCase = new CreateSessionUseCase(repo)

// CLI Controller
class CLIController {
  handleCommand(args) {
    return useCase.execute({ project: args[0] })
  }
}

// REST Controller
class APIController {
  handleRequest(req, res) {
    return useCase.execute({ project: req.body.project })
  }
}

// Desktop Controller
class ElectronController {
  handleIPC(event, data) {
    return useCase.execute({ project: data.project })
  }
}
    1. Dependency Inversion Principle Inner layers define interfaces (ports), outer layers implement them (adapters). This inverts dependencies:
    // Domain defines what it needs (port)
class ISessionRepository {
  async save(session) {
    throw new Error('Not implemented')
  }
}

// Adapters provide implementations
class FileSystemSessionRepository extends ISessionRepository {
  async save(session) {
    /* write to file */
  }
}

class DatabaseSessionRepository extends ISessionRepository {
  async save(session) {
    /* write to DB */
  }
}

// Use case doesn't care which implementation
const useCase = new CreateSessionUseCase(new FileSystemSessionRepository())
// Later, swap easily:
const useCase = new CreateSessionUseCase(new DatabaseSessionRepository())
    1. Domain-Driven Design Benefits Business rules live in entities, not scattered across layers:
    class Session {
  end() {
    // Business rule: Can only end active sessions
    if (this.state !== SessionState.ACTIVE) {

    this.endTime = new Date()

}
    1. Clear Boundaries Reduce Cognitive Load For ADHD-friendly development, clear boundaries help:
    2. "This is domain code - pure business logic, no I/O"
    3. "This is adapter code - talks to external systems"
    4. "Dependencies always point inward"

    Positive¶

    • âœ… Highly testable - Business logic has zero I/O dependencies

    Negative¶

    Neutral¶

    • â„¹ï¸ Documentation critical - Need good examples and reference cards

    Validation¶

    Success Criteria¶

    • [ ] Domain layer has zero external dependencies (pure business logic)
    • [ ] Use cases can be tested with in-memory repositories (no file I/O)

    • [ ] Adding new features takes < 30 min following established patterns
