    üß† ADHD-Friendly Workflows - Quick Wins Guide¶

    Focus: Top 10 highest-impact workflows for daily R package development

    Read Time: 5 minutes | Apply Time: Immediate Last Updated: 2025-12-21 (Updated for 28-alias system)

    üìã Table of Contents¶

    # Workflow Time Load Use When
    1 Quick Test 5 min üü¢ Made code changes
    2 Load + Test 5 min üü¢ Fresh start on package
    3 Doc + Test 8 min üü¢ Changed function docs
    4 Full Check 60 min üü° Before git commit
    5 Quick Commit 3 min üü¢ Ready to save work
    6 Fix Failing Tests varies üü° Tests are red
    7 Context Check 30 sec üü¢ Lost context
    8 Focus Mode setup üü¢ Need concentration
    9 Start Feature 2 min üü¢ Beginning new work
    10 Emergency Recovery varies üî¥ Something broke
    11 Git Feature Workflow 2 min üü¢ Feature branches
    12 Worktrees 1 min üü¢ Parallel development

    Cognitive Load: üü¢ Easy | üü° Medium | üî¥ Hard

    1Ô∏è‚É£ Quick Test Cycle¶

    When: You made code changes and want quick feedback Time: ~5 minutes | Load: üü¢ Easy | Safety: üü¢ Safe

    Commands¶

    rtest                # Run all tests

    Visual Flow¶

    Code changes ‚Üí rtest ‚Üí 2-4 min wait ‚Üí ‚úÖ Green or ‚ùå Red

    Decision Points¶

    • ‚úÖ All green ‚Üí Continue coding
    • ‚ùå Some red ‚Üí Fix and re-run rtest
    • üî¥ Many red ‚Üí Run rload then rtest to reload + test

    Pro Tips¶

    üí° Set a 5-min timer while tests run (use focus 5) üí° Tests too slow? Focus on specific test files üí° Leave tests running and switch tasks (ADHD-friendly!)

    What Could Go Wrong?¶

    • Tests hang ‚Üí Ctrl+C to cancel, check for infinite loops
    • All tests fail ‚Üí Run rpkg to verify you're in right package

    2Ô∏è‚É£ Load + Test (Atomic)¶

    When: Fresh start, want to verify everything loads and works Time: ~5 minutes | Load: üü¢ Easy | Safety: üü¢ Safe

    Commands¶

    rload && rtest       # Load then test

    Visual Flow¶

    Start work ‚Üí rload && rtest ‚Üí Package loads ‚Üí Tests run ‚Üí ‚úÖ or ‚ùå

    Why This Works (ADHD-Optimized)¶

    • Automated sequence ‚Üí No decision fatigue
    • Clear output ‚Üí Green = go, Red = stop
    • Verifies everything ‚Üí Load + tests in one go

    When to Use¶

    • üåÖ Morning start ‚Üí Verify yesterday's work still works
    • üîÑ After git pull ‚Üí Check if team changes broke anything
    • üßπ After cleanup ‚Üí Confirm nothing broke

    Pro Tips¶

    üí° Combine with coffee break - perfect timing! üí° First command of the day ritual üí° Bookmark this as your "good morning" command

    3Ô∏è‚É£ Document + Test¶

    When: You changed function documentation (roxygen comments) Time: ~8 minutes | Load: üü¢ Easy | Safety: üü¢ Safe

    Commands¶

    rdoc && rtest        # Document then test

    Visual Flow¶

    Change @param docs ‚Üí rdoc && rtest ‚Üí Docs regenerate ‚Üí Tests run ‚Üí ‚úÖ

    What This Does¶

    1. ‚è±Ô∏è [3-5s] Regenerates .Rd files from roxygen
    2. ‚è±Ô∏è [3-5s] Updates NAMESPACE exports
    3. ‚è±Ô∏è [2-4min] Runs all tests

    Common Scenario¶

    You: Added new function parameter
You: Updated @param documentation
Run: rdoc && rtest
Result: Documentation updated + tests verify it works

    Pro Tips¶

    üí° Always run after changing ANY roxygen comment üí° Catches missing @export tags early üí° Faster than full rcheck

    Safety Checks¶

    • üü¢ Safe - only regenerates docs
    • Auto-backs up NAMESPACE (devtools handles this)

    4Ô∏è‚É£ Full Check (Before Commit)¶

    When: Ready to commit, need to verify everything is perfect Time: ~60 minutes | Load: üü° Medium | Safety: üü¢ Safe

    Commands¶

    rcycle               # Full cycle: doc ‚Üí test ‚Üí check
# OR (step by step)
rdoc && rtest && rcheck

    Visual Flow¶

    Ready to commit ‚Üí rcycle ‚Üí 60 min wait ‚Üí ‚úÖ 0 errors/warnings/notes

    What This Does¶

    1. ‚è±Ô∏è [5s] Documents package
    2. ‚è±Ô∏è [2-4min] Runs tests
    3. ‚è±Ô∏è [30-60min] Full R CMD check

    ADHD Strategy for Long Wait¶

    • ‚è∞ Set timer ‚Üí focus 60 for focus timer
    • üéØ Switch context ‚Üí Work on different package
    • ‚òï Take break ‚Üí Perfect time for lunch/walk
    • üìß Other tasks ‚Üí Email, admin work

    Decision Points¶

    • ‚úÖ 0 errors, 0 warnings, 0 notes ‚Üí COMMIT! üéâ
    • ‚ö†Ô∏è Warnings/notes ‚Üí Investigate (might be OK)
    • ‚ùå Errors ‚Üí Fix, run rcheck again

    Pro Tips¶

    üí° Run this before ANY git commit üí° Use rcheckfast for quicker check (skips examples/vignettes) üí° NEVER commit with errors

    What Could Go Wrong?¶

    • Check fails ‚Üí Read error messages carefully
    • Takes forever ‚Üí Normal! R CMD check is thorough
    • Interrupted ‚Üí Just re-run to continue

    5Ô∏è‚É£ Quick Commit Workflow¶

    When: Tests pass, ready to save work to git Time: ~3 minutes | Load: üü¢ Easy | Safety: üü¢ Safe

    Commands¶

    # Option 1: Quick (if already checked)
git add . && git commit -m "message"

# Option 2: Safe (docs + tests + commit)
rdoc && rtest && git add . && git commit -m "message"

# Option 3: Ultra-safe (full check + commit)
rdoc && rtest && rcheck && git add . && git commit -m "message"

    Visual Flow¶

    Code ready ‚Üí git commit ‚Üí Done in 30s

    Commit Message Templates¶

    # Feature
git commit -m "feat: add sensitivity analysis function"

# Bug fix
git commit -m "fix: handle NA values in mediation estimate"

# Documentation
git commit -m "docs: update README with new examples"

# Tests
git commit -m "test: add tests for interaction effects"

# Refactor
git commit -m "refactor: simplify bootstrap algorithm"

    Decision Tree¶

    Did you run rcheck?
‚îú‚îÄ Yes ‚Üí git add . && git commit -m "message"
‚îú‚îÄ No ‚Üí rdoc && rtest then commit (safer)
‚îî‚îÄ Not sure ‚Üí Full check first (safest)

    Pro Tips¶

    üí° Commit often (every 30-60 min of work) üí° Small commits = easier to undo üí° Use clear messages (future you will thank you!)

    Safety Checks¶

    • Always run tests before committing
    • Can undo with git reset HEAD~1 if needed

    6Ô∏è‚É£ Fix Failing Tests¶

    When: Tests are red, need to debug Time: Varies | Load: üü° Medium | Safety: üü¢ Safe

    Step-by-Step Process¶

    Step 1: Identify the Problem (2 min)¶

    rtest                # Run tests, read error messages

    Look for:

    • Which test file failed? (test-*.R)
    • Which expectation failed? (expect_equal, etc.)
    • What was expected vs actual?

    Step 2: Run Single Test (1 min)¶

    # Run specific test file in R:
devtools::test_file("tests/testthat/test-myfunction.R")

    Step 3: Interactive Debugging (varies)¶

    rload                # Load package
# Then in R console:
# debug(myfunction)
# Run test interactively

    Step 4: Fix and Verify (2 min)¶

    # Fix the code
rtest                # Re-run all tests

    Common Test Failures¶

    1. "Error: object not found"

    • Cause: Function not exported or loaded
    • Fix: Add @export tag, run rdoc && rtest

    2. "Expected X but got Y"

    • Cause: Logic error or outdated test
    • Fix: Check function logic or update test

    3. "Test times out"

    • Cause: Infinite loop or very slow code
    • Fix: Add timeout or optimize code

    ADHD-Friendly Debug Loop¶

    1. Read error (30s)
2. Hypothesize fix (1 min)
3. Make ONE small change
4. Test immediately with `rtest`
5. Repeat until green

    Pro Tips¶

    üí° Only fix ONE test at a time üí° Use browser() for interactive debugging üí° Take breaks if frustrated

    7Ô∏è‚É£ Where Am I? (Context Check)¶

    When: Lost context, can't remember what you were doing Time: ~30 seconds | Load: üü¢ Easy | Safety: üü¢ Safe

    Commands¶

    pwd                  # Show current directory
rpkg                 # Package info & status
git status           # Check git status

    Visual Output¶

    rpkg ‚Üí Shows:
- üì¶ R package name + version
- üìä Package description
- üîß Git branch

    Quick Recovery Checklist¶

    • [ ] Run pwd ‚Üí See where you are
    • [ ] Run rpkg ‚Üí Package information
    • [ ] Run git status ‚Üí Check git status
    • [ ] Check .STATUS file for next action

    Common Scenarios¶

    "I forgot what I was working on"

    pwd                  # Current directory
rpkg                 # Package info
cat .STATUS          # Check status file

    "I don't remember what this package does"

    rpkg                 # Package info
cat DESCRIPTION      # Read full description

    "Did I make changes?"

    git status           # Git status
git diff             # See changes

    Pro Tips¶

    üí° Start every session with rpkg üí° Keep .STATUS file updated üí° Use git status frequently

    8Ô∏è‚É£ Focus Mode (Deep Work)¶

    When: Need concentration, minimize distractions Time: Setup < 1 min | Load: üü¢ Easy | Safety: üü¢ Safe

    Commands¶

    focus                # Minimize distractions
focus 25             # Focus + 25-min timer (Pomodoro)

    What focus Does¶

    1. ‚úÖ Turns off macOS notifications
    2. ‚úÖ Closes Slack, Mail, Messages
    3. ‚úÖ Starts optional timer
    4. ‚úÖ Announces when session ends

    Visual Flow¶

    Need focus ‚Üí focus 25 ‚Üí Work uninterrupted ‚Üí Timer alert ‚Üí Break

    Recommended Focus Workflows¶

    Pomodoro (25 min)

    f25                  # 25-minute focus timer
# Work for 25 min
# Timer alerts when done
# Take 5-min break

    Deep Work (50 min)

    f50                  # 50-minute deep work timer
# Work for 50 min
# Timer alerts when done
# Take 10-15 min break

    Custom Duration

    focus 90             # Custom 90-minute session
# Work until timer ends

    Pro Tips¶

    üí° Use f25 or f50 aliases for common durations üí° Use during rcheck 60-min wait üí° Take regular breaks to maintain focus

    After Focus Session¶

    # Update status and commit progress
git add . && git commit -m "Progress on X"

    9Ô∏è‚É£ Start New Feature¶

    When: Beginning new function or feature Time: ~2 minutes | Load: üü¢ Easy | Safety: üü¢ Safe

    Step-by-Step Checklist¶

    [ ] 1. Context Setup (30s)¶

    pwd                  # Verify location
rpkg                 # Check package info
git status           # Check git status

    [ ] 2. Create Function File (30s)¶

    # Create R/myfunction.R manually
# OR use usethis:
usethis::use_r("myfunction")

    [ ] 3. Create Test File (30s)¶

    # Create tests/testthat/test-myfunction.R
# OR use usethis:
usethis::use_test("myfunction")

    [ ] 4. Document Template (30s)¶

    Add roxygen skeleton:

    #' Function Title
#'
#' @param x Description
#' @return Description
#' @export
#' @examples
#' myfunction(x = 1)
myfunction <- function(x) {
  # TODO: Implement
}

    [ ] 5. First Test (30s)¶

    test_that("myfunction works", {
  result <- myfunction(x = 1)
  expect_true(!is.null(result))
})

    [ ] 6. Verify Setup¶

    rload && rtest       # Load + test
# Should load successfully, test might fail (that's OK!)

    Quick Start Template¶

    # Create files, then verify:
rload && rtest

    Pro Tips¶

    üí° Start with test first (TDD approach) üí° Make smallest working version üí° Commit early: git commit -m "feat: add myfunction skeleton"

    üîü What Did I Break? (Emergency Recovery)¶

    When: Something broke and you're not sure what Time: Varies | Load: üî¥ Hard | Safety: üü° Careful

    Emergency Triage (2 min)¶

    Step 1: Assess Damage¶

    pwd                  # Where am I?
git status           # What changed?
rtest                # Do tests pass?

    Step 2: Identify Problem¶

    Tests failing?

    rtest                # Run tests
# Read error messages carefully
# Jump to workflow #6 (Fix Failing Tests)

    Package won't load?

    rload                # Try to load
# Read error messages
# Common: syntax error, missing dependency

    Git issues?

    git status           # Git status
git log --oneline -5 # Recent commits

    Recovery Options (Choose One)¶

    Option A: Recent Changes (Most Common)¶

    # Undo last commit (keeps changes)
git reset HEAD~1
# Fix the issue
rtest                # Verify tests pass
git add . && git commit -m "fix: ..."

    Option B: Code Error¶

    # Use editor to fix syntax error
# Then:
rload                # Try loading again
rtest                # Run tests

    Option C: Clean Build (Last Resort)¶

    # Clean and rebuild
rm -rf man/ NAMESPACE
rdoc && rtest        # Regenerate docs + test

    Prevention Checklist¶

    • ‚úÖ Run tests before commits
    • ‚úÖ Commit frequently (small changes)
    • ‚úÖ Use version control
    • ‚úÖ Test after major changes

    Pro Tips¶

    üí° Don't panic - almost everything is reversible üí° Read error messages slowly (pause before acting) üí° Ask for help: cc (Claude Code) or colleagues

    When to Ask for Help¶

    • üî¥ Spent > 30 min stuck
    • üî¥ Don't understand error message
    • üî¥ Afraid of making it worse

‚îÇ  ‚îú‚îÄ Did full check? ‚Üí git add . && git commit -m "msg"

‚îÇ
‚îú‚îÄ Tests are failing
‚îÇ  ‚îî‚îÄ See workflow #6 (Fix Failing Tests)

‚îÇ  ‚îî‚îÄ Run: rpkg

    ‚è±Ô∏è Time-Based Quick Reference¶

    1Ô∏è‚É£1Ô∏è‚É£ Git Feature Branch Workflow (v4.1.0+)¶

    When: Working on new features, hotfixes, or any code changes Time: ~2 minutes | Load: üü¢ Easy | Safety: üü¢ Safe

    Commands¶

    # Start a feature
g feature start auth-improvements

# Work on your feature...
g add . && g commit "Add OAuth2 support"

# Stay synced with dev
g feature sync

# Finish: push + create PR
g feature finish

    Visual Flow¶

    Start ‚Üí feature start ‚Üí code ‚Üí commit ‚Üí sync ‚Üí finish ‚Üí PR ‚Üí merge

    Quick Reference¶

    Command What It Does
    g feature start <name> Create feature branch from dev
    g feature sync Rebase feature onto dev
    g feature list List feature/hotfix branches
    g feature finish Push + create PR to dev
    g promote Create PR: feature ‚Üí dev
    g release Create PR: dev ‚Üí main
    g feature prune Delete merged branches

    Workflow Guard¶

    The workflow guard blocks direct pushes to protected branches:

    # Blocked ‚ùå
git checkout main && g push   # Direct push to main blocked

# Allowed ‚úÖ
g feature start my-fix        # Create feature branch
g push                        # Push feature branch OK
g promote                     # Create PR instead

    Pro Tips¶

    üí° Run g feature sync daily to avoid merge conflicts üí° Use g feature finish to push + create PR in one step üí° Clean up with g feature prune weekly
