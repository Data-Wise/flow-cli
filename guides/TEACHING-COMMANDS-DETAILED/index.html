    Teaching Commands - Detailed Guide¶

    Version: 1.0 Status: Production Ready Last Updated: 2026-01-13

    Overview¶

    This guide provides comprehensive, step-by-step explanations of each teaching command, what it does, why you'd use it, and real-world workflows.

    Command Breakdown¶

    1. teach init - Initialize a New Course¶

    What it does: Sets up a complete teaching workflow for a course repository. This one-time setup creates all the infrastructure needed for course website management.

    Syntax: 

    teach init "Course Name"              # Interactive mode
teach init -y "Course Name"           # Non-interactive (safe defaults)
teach init --dry-run "Course Name"    # Preview without changes

    When to use: - First time setting up a course repository - Migrating an existing course to flow-cli - Setting up a new semester variant of a course

    What happens step-by-step:

    1. Configuration Creation
    2. Creates .flow/teach-config.yml configuration file
    3. Stores course name, semester, schedule, and branch names

    4. Example: 

      course:
  name: "STAT 545"
  semester: "spring"
  year: 2026

branches:
  draft: "draft"           # Where you make edits
  production: "production" # What students see

semester_info:
  start_date: "2026-01-13"
  end_date: "2026-05-08"

    5. Branch Setup

    6. Creates draft and production branches
    7. draft: Your working branch (safe to edit)
    8. production: Student-facing branch (auto-deployed)

    9. Sets draft as default branch

    10. Script Installation

    11. Creates scripts/quick-deploy.sh - One-command deployment
    12. Creates scripts/semester-archive.sh - End-of-semester archival

    13. Both scripts are executable and ready to use

    14. GitHub Actions Setup

    15. Creates .github/workflows/ if needed
    16. Configures automatic deployment to GitHub Pages
    17. Triggers on production branch updates

    18. Students see updates automatically

    19. Git Commit

    20. Stages all new files
    21. Creates initial commit with setup
    22. Ready for pushing to GitHub

    Options:

    -y or --yes (Non-interactive mode): - Accepts safe defaults without prompting - Great for automated setup or when you know what you want - Defaults: - ‚úÖ Keep git history (in-place conversion) - ‚úÖ Auto-exclude build artifacts (renv, _build, etc) - ‚úÖ Use sensible semester dates - ‚úÖ Skip GitHub push (do manually)

    --dry-run: - Shows what would happen without making changes - Preview the setup plan - Useful to see what files will be created

    Interactive prompts: - Course name (already provided) - Semester and year - Branch names (usually draft and production) - Semester start/end dates - Include breaks? (Spring Break, etc) - Push to GitHub now? (optional)

    Example:

    # Initialize a new course - interactive
cd ~/projects/teaching/stat-545
teach init "STAT 545"

    2. teach deploy - Deploy Changes to Production¶

    What it does: Moves your changes from the draft branch (where you edit) to the production branch (what students see). Safely merges, handles conflicts, and triggers GitHub Pages deployment.

    Syntax: 

    teach deploy

    When to use: - After making changes to lectures, assignments, solutions - Publishing new course materials - Fixing typos on the live site - Multiple times per day (very fast!)

    What happens step-by-step:

    1. Safety Checks
    2. Verifies you're on draft branch
    3. Checks for uncommitted changes
    4. Ensures no conflicts with production

    5. Prevents accidental overwrites

    6. Branch Merge 

      draft (your edits)
   ‚Üì
production (students see this)
   ‚Üì
GitHub
   ‚Üì
GitHub Pages (automatic deployment)
   ‚Üì
Live website

    7. Git Operations

    8. Commits any staged changes
    9. Merges draft ‚Üí production
    10. Returns to draft branch

    11. Shows merge summary

    12. Deployment Trigger

    13. Pushes to GitHub
    14. GitHub Actions workflow starts automatically
    15. Site rebuilds and deploys to GitHub Pages

    16. Usually takes 1-2 minutes

    17. Completion Status

    18. Shows successful deployment message
    19. URL to live site
    20. Suggestion to verify changes

    Example: Daily Typo Fix

    # You notice a typo on the live site
cd ~/projects/teaching/stat-545
work stat-545          # Starts work session on draft branch

# Make edits...
# Fix: "Lineare regression" ‚Üí "Linear regression" in lectures/week05.qmd

git add lectures/week05.qmd
git commit -m "fix: correct spelling in week 5"

# Deploy immediately
teach deploy

    3. teach status - View Course Status¶

    12. Last commit message

    13. Last commit time

    14. Safety Indicators

    15. üü¢ Safe (on draft branch)
    16. üü° Warning (on production branch)

    17. üî¥ Needs attention (uncommitted changes)

    What it does: Displays the current week number in your semester, useful for checking course progress and scheduling.

    Syntax:

    5. teach archive - Archive a Semester¶

    What it does: Creates a permanent snapshot of your semester at its completion. Useful for: - Preserving final state of course - Starting fresh for next semester - Comparing changes between years - Regulatory/institutional requirements

    Syntax: 

    teach archive

    When to use: - End of each semester (after final grades submitted) - Before starting a new semester variant - Creating historical records - Backing up course state

    What happens:

    1. Validation
    2. Confirms you want to archive
    3. Shows current semester info

    4. Lists files that will be tagged

    5. Git Tag Creation

    6. Creates annotated git tag: spring-2026-final
    7. Captures complete repository state

    8. Includes timestamp and notes

    9. Configuration Update Prompt

    10. Shows instructions for next semester
    11. Suggests: Update course.year and course.semester

    12. Points to .flow/teach-config.yml

    13. Backup Created

    14. Tag is pushed to GitHub
    15. Immutable snapshot accessible forever
    16. Can browse entire codebase at that point

    Example workflow:

    # End of semester
teach archive

    6. teach config - Edit Configuration¶

    What it does: Opens the .flow/teach-config.yml file in your editor for customization.

    Syntax: 

    teach config

    When to use: - Updating semester dates - Changing course name or code - Modifying branch names - Adding break periods - Configuring shortcuts

    Configuration file reference:

    # Basic course info
course:
  name: "STAT 545"           # Full course name
  code: "stat-545"           # Short code (used for shortcuts)
  semester: "spring"         # spring|summer|fall|winter
  year: 2026
  full_name: "Statistical Methods I"
  instructor: "Your Name"

# Semester schedule
semester_info:
  start_date: "2026-01-13"   # YYYY-MM-DD
  end_date: "2026-05-08"
  breaks:
    - name: "Spring Break"
      start: "2026-03-10"
      end: "2026-03-17"
    - name: "Reading Week"
      start: "2026-04-14"
      end: "2026-04-18"

# Git branch names
branches:
  draft: "draft"             # Your working branch
  production: "production"   # Students see this

# Custom shortcuts (auto-loaded in work sessions)
shortcuts:
  s545: "work stat-545"      # Quick work shortcut
  s545d: "./scripts/quick-deploy.sh"  # Quick deploy

    Common edits:

    After semester ends: 

    # OLD
semester: "spring"

    3. Sets up question structure

    4. Configures scoring and timing

    5. Interactive Prompts (if using examark)

    6. Exam name / title

    9. Question types

    10. Output Format

    11. Human-readable markdown
    12. Machine-readable for grading
    13. Canvas QTI compatible (for upload)

    Troubleshooting¶

    "Must be on draft branch" error¶

    Problem: You tried to deploy but you're on production branch

    Solution: 

    # Check current branch
git branch --show-current

# Switch to draft
git checkout draft

# Now try deploy
teach deploy

    Prevention: Work sessions automatically warn if on production

    "Uncommitted changes" warning¶

    Problem: You made edits but forgot to commit

    Solution: 

    # Check what changed
git status

# Stage changes
git add .

# Commit with message
git commit -m "fix: update lecture notes"

# Now deploy
teach deploy

    Deploy seems slow¶

    Problem: Deployment taking > 3 minutes

    Check: 

    # View GitHub Actions status
gh run list

# Or visit GitHub Actions tab:
# github.com/your-org/repo-name/actions

    Usually caused by: - Large file changes (PDFs, images) - First-time GitHub Pages setup - GitHub service delays (rare)
